# پلن اجرایی آماده‌سازی محصول برای کاربر نهایی

این سند **ترتیب اجرا و برید باریک** را برای رسیدن سریع‌تر به «قابل استفادهٔ واقعی توسط کاربر نهایی» مشخص می‌کند. **مرز محصول و فهرست v1** در [`PLATFORM_GO_LIVE_ROADMAP.md`](./PLATFORM_GO_LIVE_ROADMAP.md) است؛ اینجا فقط **چگونگی هم‌زمان کردن تیم‌ها** و **مسیر بحرانی** آمده است.

**بک‌لاگ اجرایی (استوری، مالک نقش، DoD، وابستگی):** [`GO_LIVE_SPRINT_BACKLOG.md`](./GO_LIVE_SPRINT_BACKLOG.md)

---

## ۱) اصول تصمیم‌گیری (برای سرعت بدون فروپاشی)

| اصل | معنی عملی |
|-----|-----------|
| **یک مرجع محصول** | هر چیزی خارج از جدول v1 در [`PLATFORM_GO_LIVE_ROADMAP.md`](./PLATFORM_GO_LIVE_ROADMAP.md) بدون تأیید محصول وارد اسپرینت نشود. |
| **Staging = حقیقت** | تا وقتی روی staging با API واقعی سبز نشده، «آمادهٔ کاربر» اعلام نشود؛ mock/dev-bypass فقط توسعه است. |
| **بک‌اند جلوتر از UI تزئینی** | قرارداد `/api/v1`، دادهٔ PostgreSQL، RBAC سمت سرور؛ UI فقط انطباق و حالت خالی/خطا. |
| **یک برید باریک** | در هر اسپرینت حداکثر **۲–۳ جریان کاربر کامل** از ورود تا نتیجه (مثلاً OTP → داشبورد → یک قرارداد). |
| **اندازه‌گیری** | برای هر جریان: زمان پاسخ p95، نرخ خطا، و یک شمارندهٔ «موفقیت task» در analytics (حتی حداقلی). |

## ۲) لایهٔ زیرساخت فنی (در مخزن؛ پایهٔ بقیهٔ فازها)

این بخش **کاهش اصطکاک توسعه** است؛ به‌تنهایی کاربر نهایی نمی‌سازد ولی باعث می‌شود تیم روی محصول متمرکز شود.

| موضوع | وضعیت / مرجع |
|--------|----------------|
| مونوریپو، `@amline/ui-core`، `ensure-ui-core` | [`MONOREPO.md`](./MONOREPO.md) |
| قرارداد OpenAPI + تایپ TS + CI drift | [`FRONTEND_API_INTEGRATION.md`](./FRONTEND_API_INTEGRATION.md)، job `openapi-contract` در CI |
| هم‌ترازی `X-Request-Id` (fetch + axios) | همان سند، کلاینت‌های ادمین/مشاور |
| Storybook برای کامپوننت مشترک | `npm run storybook:ui-core` از ریشه |
| مرجع env | [`ENV_MATRIX.md`](./ENV_MATRIX.md) |

**اقدام باقی‌ماندهٔ پرارزش در این لایه (موازی با فاز محصول):** مهاجرت **تدریجی** تماس‌های `axios` به الگوی `fetchJson` در مسیرهای پرتکرار (کاهش دو مسیره بودن خطا).

## ۳) مسیر بحرانی تا اولین کاربر معتبر (خلاصهٔ فازها)

```mermaid
flowchart LR
  A[فاز0 قفل محصول و مالکیت] --> B[فاز1 استیجینگ و DB واقعی]
  B --> C[فاز2 هسته v1 روی staging]
  C --> D[فاز3 امنیت و نشست پرود]
  D --> E[فاز4 استقرار و رصد]
  E --> F[فاز5 اعتماد و پشتیبانی]
```

| فاز | هدف | خروجی قابل سنجش | وابستگی |
|-----|-----|------------------|---------|
| **۰ — قفل و نقشه** | پر کردن مالکیت و تأیید جدول v1 | نام تیم‌ها در roadmap؛ توقف scope creep | — |
| **۱ — استیجینگ** | محیط شبیه prod، Postgres، بدون MSW/bypass | URL staging؛ قرمز/سبز health | DevOps + Backend |
| **۲ — هسته v1** | همهٔ ردیف‌های «داخل v1» با دادهٔ واقعی | چک‌لیست انتهای [`PLATFORM_GO_LIVE_ROADMAP.md`](./PLATFORM_GO_LIVE_ROADMAP.md) | Backend + Frontend + QA |
| **۳ — امنیت پرود** | OTP، کوکی httpOnly، rate limit، RBAC enforce | بدون dev bypass در بیلد؛ تست نفوذ سبک | Backend + Security |
| **۴ — استقرار** | Production جدا، CI سبز، Sentry/لاگ | Deploy تک‌کلیک یا runbook؛ correlation id | DevOps |
| **۵ — اعتماد** | بک‌آپ، پیام خطای شفاف، SLA داخلی | runbook بازیابی؛ پشتیبانی آماده | همه |

جزئیات فازهای ۰–۵ در همان [`PLATFORM_GO_LIVE_ROADMAP.md`](./PLATFORM_GO_LIVE_ROADMAP.md) آمده؛ اینجا **فقط ترتیب و تمرکز** است.

## ۴) برنامهٔ زمانی پیشنهادی (چابک؛ قابل تنظیم با ظرفیت تیم)

| بازهٔ زمانی | تمرکز اصلی | خروجی حداقلی |
|-------------|------------|---------------|
| **هفته ۱** | فاز ۰ + بالا آوردن staging + مهاجرت اولین APIهای بحرانی از mock | استیجینگ پایدار؛ ۱ جریان E2E روی staging (مثلاً login → dashboard) |
| **هفته ۲–۳** | فاز ۲: قرارداد، CRM، کاربران طبق جدول v1 | هر هفته ۲–۳ مسیر admin سبز در staging + تست دستی ثبت‌شده |
| **هفته ۴** | فاز ۳ + سخت‌سازی env پروداکشن | بیلد admin بدون bypass؛ چک‌لیست امنیت |
| **هفته ۵** | فاز ۴ + فاز ۵ سبک | اولین کاربر محدود (beta) با نظارت و رول‌بک آماده |

اگر تیم کوچک است: **فقط یک سطح کاربر** (مثلاً فقط ادمین داخلی) را تا «پایدار» برسانید؛ `amline-ui` و مشاور را موازی ولی **بعد از** ثبات API قرار دهید تا دوباره‌کاری نشود.

## ۵) مسیرهای موازی (بدون انحراف از v1)

| مسیر | توضیح |
|------|--------|
| **اپ کاربر (`amline-ui`)** | بعد از ثبات auth و قرارداد روی staging؛ E2E موجود را گسترش دهید. |
| **مشاور (`consultant-ui`)** | وقتی `/consultant/*` روی backend واقعی با همان قرارداد خطا آماده شد. |
| **سایت مارکتینگ (`site`)** | مستقل از API اصلی؛ می‌تواند موازی برود. |
| **Hamgit / Market** | طبق roadmap عمداً **بعد از v1**. |

## ۶) تعریف «آماده برای کاربر نهایی» (DoD)

حداقل قبل از باز کردن ثبت‌نام عمومی یا فروش:

1. چک‌لیست **«قبل از اولین کاربر واقعی»** در [`PLATFORM_GO_LIVE_ROADMAP.md`](./PLATFORM_GO_LIVE_ROADMAP.md) همه تیک خورده باشد.
2. **بدون** `VITE_ENABLE_DEV_BYPASS` و **بدون** MSW در buildهای استقرار یافته.
3. حداقل یک **دور بازیابی** از بک‌آپ DB تست شده (حتی روی استیج).
4. مسیر پشتیبانی (ایمیل/تیکت) و **پاسخ‌گویی** تعریف شده.

برای **بتا محدود** می‌توان DoD را سبک‌تر کرد (مثلاً فقط زیرمجموعهٔ v1)؛ باید **صریح** در محصول نوشته شود.

## ۷) ریسک‌ها و کاهش آن

| ریسک | کاهش |
|------|------|
| اختلاف mock و backend | OpenAPI + CI drift؛ تست روی staging نه فقط mock. |
| scope انفجار | جدول v1؛ هر فیچر جدید = تأیید محصول. |
| امنیت OTP/نشست | فاز ۳ قبل از ترافیک واقعی؛ rate limit. |
| فرانت دو شاخه (axios/fetch) | مهاجرت تدریجی به یک الگوی خطا و `request_id`. |
