# نگاشت فیچرهای اپ کاربری املاین (`app.amline.ir`) و پروژه

## محدودیت بررسی زنده

درخواست HTTP مستقیم به `https://app.amline.ir/` از محیط build ابزار **timeout** شد؛ بنابراین **استخراج زندهٔ DOM/مسیرها** ممکن نشد. این سند بر پایهٔ موارد زیر تهیه شده است:

- معرفی محصول و مراحل قرارداد در **وب‌سایت رسمی املاین** (`amline.ir`): رهن‌واجاره و خرید‌وفروش، احراز هویت، ثبت ملک/اسناد، مبالغ و پرداخت، امضای دیجیتال، تأیید کارشناس حقوقی، کد رهگیری، پشتیبانی حقوقی/داوری.
- ریپوی **`amline-ui`** (پنل کاربری نمونه در همین مونورپو): ورود OTP، لیست قراردادها، ویزارد انعقاد، جزئیات قرارداد.
- ریپوی **`admin-ui`**: پنل مدیریت، ویزارد ادمین، CRM، کاربران پیشرفته، قراردادها، ممیزی و غیره.

برای **تطبیق ۱۰۰٪** با اپ production، یک بار با مرورگر وارد `app.amline.ir` شوید و این جدول را با منوی واقعی به‌روز کنید.

## فیچرهای محصول (کاربر نهایی) — چک‌لیست

| حوزه | توضیح کوتاه | نزدیک‌ترین پیاده‌سازی در ریپو |
|------|-------------|--------------------------------|
| ورود با موبایل / OTP | احراز ورود | `amline-ui` `/login`؛ `admin-ui` ورود آزمایشی + `/auth/me` |
| داشبورد / قراردادهای من | لیست و وضعیت | `amline-ui` `/contracts`؛ `admin-ui` `/contracts` |
| انعقاد قرارداد (ویزارد) | مراحل طرفین، ملک، امضا، شاهد، … | `amline-ui` `/contracts/wizard`؛ `admin-ui` ویزارد کامل‌تر با چند Step |
| انواع قرارداد | رهن‌واجاره، خرید‌وفروش | هر دو؛ خرید‌وفروش ممکن است پشت فلگ env باشد |
| احراز هویت طرفین | KYC / مدارک | در اپ واقعی؛ در ادمین: **کاربران → احراز دستی** (MSW + UI) |
| پرداخت / کارمزد | درگاه، کمیسیون | ویجت‌های داشبورد ادمین؛ قرارداد `PENDING_COMMISSION`؛ API کمیسیون در mock |
| امضای دیجیتال | مرحله Signing | `SigningStep` در admin ویزارد |
| شاهد | Witness + OTP mock | `WitnessStep` |
| تأیید ادمین / حقوقی | صف قراردادهای در انتظار | `admin-ui` `/contracts` با `PENDING_ADMIN_APPROVAL` + تأیید/رد |
| کد رهگیری | پس از تکمیل رسمی | فیلد **`tracking_code`** در لیست/جزئیات (mock + UI) |
| وضعیت بررسی حقوقی | کارشناس حقوقی | فیلد **`legal_review_status`** در mock + ستون ادمین |
| کیف پول کاربر | موجودی، تراکنش | پروفایل کاربر در ادمین + `financials/wallets` mock |
| پشتیبانی / تیکت | درخواست کاربر | تب تیکت در جزئیات کاربر |
| اعلان‌ها | inbox | `/admin/inbox` + زنگوله |
| CRM / لید | فروش سازمانی | `admin-ui` CRM (Kanban، جزئیات، bulk) |

## شکاف‌های معمول (بین اپ کاربر و ادمین)

1. **دادهٔ واقعی**: همهٔ موارد بالا در MSW نمونه‌اند؛ API production باید همان قرارداد را برگرداند.
2. **فلگ محصول**: مسیرها یا انواع قرارداد ممکن است با `VITE_*` در اپ کاربر و ادمین متفاوت کنترل شوند؛ مستند env را یکسان نگه دارید.
3. **کد رهگیری و حقوقی**: تا وقتی backend فیلد ندهد، فقط در mock و UI ادمین دیده می‌شود.

## اقدامات پیشنهادی بعدی

- اتصال `GET /contracts/list` واقعی به فیلتر `status` / `type` / صفحه‌بندی (در mock اعمال شده است).
- همسان‌سازی پاسخ `contracts/list` بین **اپ کاربر** و **ادمین** (همیشه `{ items, total, page, limit }` یا هر دو شکل با adapter).
- مستند کردن endpointهای «کد رهگیری» و «تأیید حقوقی» در OpenAPI بک‌اند.

آخرین به‌روزرسانی این سند: هم‌زمان با اضافه شدن فیلدهای نمونهٔ قرارداد و فیلتر MSW در `admin-ui`.
